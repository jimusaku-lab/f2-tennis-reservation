Version: 0.1
Date: 2026-06-07
Related: ./design.md

## 1. 実装の前提

本アプリは、F2テニス専用の参加予約システムである。実装上の最重要方針は以下の3点。

- 広告なし
- パスワードなし
- 参加予約が10秒で終わる

初期MVPでは、サークルスクエアのような汎用グループウェアを再現しない。対象は、予定確認、回答期限表示、参加予約、不参加・未定回答、キャンセル、定員管理、キャンセル待ち、参加状況集計、回答者リスト、メンバー自己申請、管理者承認、必要時のメンバー招待、管理者による予定作成・編集・削除に絞る。

表示名:

- サークル名: F2テニス
- ログイン画面・サービス名: F2テニス参加予約
- アプリヘッダー・管理画面名: F2テニス予約管理

想定技術構成:

- Frontend: React + Vite + TypeScript
- Styling: CSS Modules または Tailwind CSS
- Backend/Auth/DB: Supabase
- Hosting: GitHub Pages
- Repository: GitHub

## 5. Phase 2: Supabase基盤

### 作業内容

- Supabase Authを有効化
- メールOTPまたはマジックリンクを設定
- PostgreSQLテーブルを作成
- Row Level Securityを有効化
- 初期RLS policyを作成
- ローカルからSupabaseへ接続するクライアントを実装

### 作成テーブル

- `members`
- `events`
- `reservations`
- `invitations`
- `notifications`

### 主要制約

- `members.auth_user_id` はSupabase Auth user IDと紐づく
- `reservations` は `event_id + member_id` で一意
- `events.capacity` は1以上
- `events.response_deadline` は回答期限を表す
- `reservations.status` は `confirmed`, `declined`, `tentative`, `waitlisted`, `cancelled`
- `events.status` は `draft`, `published`, `cancelled`, `deleted`
- `events.deleted_at` と `events.deleted_by` を持たせ、削除はソフト削除で実装する

### 完了条件

- DB migration SQLが保存されている
- RLSが有効である
- 非ログインユーザーはデータを取得できない
- 一般メンバーは自分の予約だけ作成・変更できる
- 管理者は予定とメンバーを管理できる

## 6. Phase 3: メンバー認証

### 作業内容

- ログイン画面を実装
- メールOTPまたはマジックリンク認証を実装
- ログイン後に `members` レコードを取得
- 承認済みでないユーザーをブロック
- 公開URLからの利用申請フォームを実装
- 管理者の申請承認・却下処理を実装
- `disabled` メンバーをブロック
- セッション保持を確認
- 招待メール送信処理を実装
- メール未設定のデモ環境では、招待メールが送られないことを管理者画面に明示

### 推奨仕様

- ログイン画面ではメールアドレスだけ入力する
- パスワード入力欄は作らない
- ログイン後は予定一覧へ遷移する
- ログイン状態が有効なら直接予定一覧を表示する
- 初回登録または招待承認時に、表示名またはニックネームを入力・確認できる
- 表示名またはニックネームは、回答者リストや参加者一覧に表示される
- F2テニスメンバーへの通常告知は、管理者がアプリURLを共有し、メンバーが自己申請する
- 自己申請直後のユーザーは承認待ち画面を表示し、予定一覧・予約機能には進ませない
- 管理者は申請一覧から承認・却下できる
- 本番環境では、管理者が招待したメールアドレスへ招待メールを送信する
- 招待メールには招待リンク、有効期限、初回登録案内を含める
- 招待メール送信に失敗した場合は、管理者画面にエラーを表示する

### 完了条件

- パスワードなしでログインできる
- 未承認ユーザーは予定一覧・予約機能を利用できない
- メンバーが公開URLから利用申請できる
- 管理者が申請を承認・却下できる
- 無効化済みユーザーは利用できない
- リロードしてもログイン状態が維持される
- 本番環境で招待メールが送信される
- デモ環境で招待メールが送信されない場合、その旨が画面上で分かる

## 8. Phase 5: 予約・キャンセル

### 作業内容

- 参加予約処理を実装
- 不参加回答処理を実装
- 未定/その他回答処理を実装
- キャンセル処理を実装
- 予約状態の即時反映
- 二重予約防止
- 定員チェック
- エラー表示

### 参加予約仕様

- 空きがあれば `confirmed`
- 定員に達していれば `waitlisted`
- 既に予約がある場合は新規作成ではなく状態更新
- 回答期限を過ぎた予定は予約不可
- 中止予定は予約不可

### 不参加・未定仕様

- 不参加は `declined`
- 未定/その他は `tentative`
- `confirmed` から `declined` または `tentative` に変わる場合は、キャンセル待ち繰り上げ処理を呼ぶ
- 出欠メモを任意で保存できる
- 更新日時を回答者リストに表示する

### キャンセル仕様

- `confirmed` または `waitlisted` を `cancelled` にする
- キャンセル後、予定一覧に即反映する
- `confirmed` のキャンセル時はキャンセル待ち繰り上げ処理を呼ぶ

### 完了条件

- 一覧から1タップで参加予約できる
- 一覧から1タップでキャンセルできる
- 詳細画面から不参加・未定/その他を回答できる
- 定員を超えてconfirmedにならない
- 同じ予定に同じメンバーが複数回答されない

## 9. Phase 6: キャンセル待ち

### 作業内容

- 定員到達時にキャンセル待ち登録
- キャンセル待ち順を管理
- confirmedキャンセル時の自動繰り上げ
- 繰り上げ通知の土台を実装

### 繰り上げルール

- `waitlisted` の中で `created_at` が最も早い予約を選ぶ
- その予約を `confirmed` に変更する
- `waitlist_position` を再計算する
- 対象メンバーに通知レコードを作成する

### 完了条件

- 定員超過時はwaitlistedになる
- キャンセル時に先着順で繰り上がる
- 繰り上がったメンバーの画面表示がconfirmedになる
- キャンセル待ち人数が正しく表示される

### 予定削除仕様

- adminのみ予定を削除できる
- 削除は物理削除ではなく、`events.status = 'deleted'` にする
- 削除時に `deleted_at` と `deleted_by` を記録する
- 削除済み予定はメンバー向け予定一覧、予定詳細、自分の予約一覧に表示しない
- 削除済み予定では新規回答・回答変更を不可にする
- 参加者またはキャンセル待ちがいる予定を削除する場合は、確認ダイアログで強く警告する
- メンバーへ中止を知らせたい場合は削除ではなく予定中止を使う

### P1: GitHub Pagesルーティング修正

対応要件:

- GitHub Pages上で `/events`、`/events/:eventId`、`/admin` などを直接開いても404にならない
- メールログイン後のリダイレクト先が正しく開く
- project pagesの場合、`VITE_APP_BASE_PATH` を `/<repository-name>/` に設定する
- ビルド後の `index.html` でasset URLが公開パスと一致する

実装方針:

- `HashRouter` に切り替える場合は、ログインリダイレクトも `/#/events` 形式に統一する
- `BrowserRouter` を維持する場合は、GitHub Pages用の `404.html` フォールバックを生成する
- どちらを採用する場合も、READMEに公開時の環境変数設定を記載する

### P2: 回答期限フォールバック修正

対応要件:

- 新規予定作成時の回答期限初期値は開始日の2日前22:00
- `response_deadline` がnullの場合の補完値も開始日の2日前22:00
- `response_deadline ?? starts_at` のように開始時刻を期限扱いしない
- フロントエンドの表示・締切判定、Supabase SQL関数、デモデータで同じルールにする

### P3: 予定詳細UI修正

対応要件:

- confirmedの場合、予定詳細の回答パネルに「参加する」を表示しない
- waitlistedの場合、予定詳細の回答パネルに「参加する」または「キャンセル待ち」を表示しない
- confirmed/waitlistedでは状態表示とキャンセル操作を主UIにする
- 未回答、declined、tentative、cancelledの場合だけ参加操作を表示する

### P3: デモデータ修正

対応要件:

- デモ予定の回答期限を開始日の2日前22:00にする
- デモ画面のサービス名を「F2テニス参加予約」にする
- デモ画面のヘッダー/管理画面名を「F2テニス予約管理」にする

### 完了条件

- F2テニス表記が全画面で統一されている
- GitHub Pages上で直URLアクセスとリロードができる
- メールログイン後のリダイレクトが本番URLで動く
